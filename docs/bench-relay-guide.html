Driving the relays before the bike

Get the relay-control firmware onto an ESP32 and click the relays on the bench — before anything is installed on the motorcycle.

This is firmware Milestone 2 in practice: a RelayController that owns the four ULN2803-driven coil enables, plus a hand-driven serial/Telnet console to actuate them. It assumes nothing about the bike — you can do all of this with just an ESP32 dev board on a USB cable, and add the relay box later. Physical, hands-on steps (probing pins, listening for clicks, all wiring) are yours; everything else runs from the firmware.

▶ Why bench-first The project rule is “every change is proven on the bench twin before it touches the bike.” This guide is that proof step for relay control.

0What you need

Minimum (logic only):

  • An ESP32 dev board (this project targets ESP32-WROOM-32D) and a USB cable.
  • A PC with VS Code + the PlatformIO IDE extension.
  • The Silicon Labs CP210x USB-to-UART driver — the boards use a CP2102, and without the driver Windows shows no COM port.

To see real relays move (optional, add when ready):

  • The pre-wired 4-relay / 6-fuse box.
  • A ULN2803A Darlington array (the coil driver — see ADR-0003).
  • A bench 12 V supply for the coil rail.
  • A multimeter, and optionally a 3.3 V LED + resistor to watch a single channel.

1Get the firmware and configure secrets

Open the firmware/ folder directly in VS Code (PlatformIO expects platformio.ini at the root of the opened folder — open firmware/, not the repo root). Copy the secrets template and fill it in (it’s gitignored, so your credentials never reach the repo):

cp include/secrets.h.example include/secrets.h

Set WIFI_SSID, WIFI_PASS, and an OTA_PASS. WiFi is optional for the bench — the relay console runs over USB whether or not WiFi connects — but valid credentials also give you the Telnet console and wireless (OTA) updates. The OTA_PASS must match the --auth= value in platformio.ini. Avoid % and $ in the OTA password (the config parser treats them specially).

2Build and flash over USB

Plug the board in. From the firmware/ directory:

pio run -e bench                 # compile
pio run -e bench -t upload       # flash over USB (auto-detects the port)

If PlatformIO picks the wrong port, name it explicitly — find the COM port in Windows Device Manager under Ports (COM & LPT) (it shows as Silicon Labs CP210x):

pio run -e bench -t upload --upload-port COM4

A successful flash ends with Hard resetting via RTS pin... and [SUCCESS].

▶ Second unit The bike ESP uses the same code — swap bench → bike. The only difference is DEVICE_NAME, set per environment in platformio.ini.

3Open the console

pio device monitor               # 115200 baud

On boot you should see something like:

=== vstrom-bench booting ===
WiFi: connecting to "your-wifi" in the background...

=== moto-11 bench relay console ===
  status | s        show every channel + GPIO + state
  on   <ch|all>      energise a channel (relay pulls in)
  off  <ch|all>      de-energise a channel
  toggle <ch>        flip a channel
  selftest           click each relay in turn (300ms each)
  help | ?           this list
  channels: master  fog  grip  spare
(onboard LED: heartbeat when all off, solid ON when any relay is on)
Bench relay console live on USB serial. Type 'help'.
WiFi OK   IP 192.168.0.104   host vstrom-bench.local — OTA + telnet console live

The console works immediately on USB. If WiFi connects, the last line appears and you also get the same console over Telnet (port 23) at the shown IP or vstrom-bench.local.

4Drive the relays

Type commands and press Enter:

CommandEffect
status (or s)List every channel, its GPIO, and ON/off state
on fogEnergise the fog channel (GPIO goes HIGH → relay pulls in)
off fogDe-energise it
toggle gripFlip the grip channel
on all / off allEnergise / de-energise all four
selftestClick each relay in turn — master, fog, grip, spare — 300 ms each, then all off
help (or ?)Reprint the command list

The four logical channels and their pins (full rationale in ADR-0009):

ChannelGPIODrives
master13External master relay — gates the whole box
fog25Fog lights
grip26Heated grips (future load)
spare27Reserved aux channel

5Confirm it actually works (your hands)

A firmware “upload OK” is not proof a relay moved. Confirm physically — pick the level of hardware you have on the bench:

A. ESP board only (no ULN/relay box yet)

The channels are real 3.3 V logic outputs but drive nothing visible. Two checks:

  • Onboard LED: idle shows a ~1 Hz heartbeat. Type on master → the LED goes solid; off all → the heartbeat returns. That proves the control logic runs.
  • Multimeter on the pin: black probe to GND, red probe to GPIO 13 / 25 / 26 / 27. It should read ~3.3 V when that channel is ON and ~0 V when off. (Optionally put a 3.3 V LED + ~330 Ω resistor from the pin to GND to watch one channel.)

B. ULN2803 wired, no relay box

Wire ULN inputs 1–4 to GPIO 13/25/26/27 in that order, ULN GND (pin 9) to the ESP GND. With a channel ON, the matching ULN output sinks to ~0 V (it’s an open-collector low-side switch); off, it floats. Confirm with the multimeter.

C. Full chain (ULN + relay box + 12 V)

Connect per the wiring schematic ULN2803 detail: the box’s commoned pin 86 is the +12 V coil rail, ULN COM (pin 10) → +12 (internal flyback), each white pin-85 coil wire sunk by one ULN output. Now each on <channel> should produce an audible relay click, and selftest clicks all four in sequence. Verify continuity across the relay’s switched contacts (30↔87) closes when energised.

⚠ Fail-safe to expect Every relay is OFF at boot and stays off until you command it — by design. Accessories are fail-OFF (ESP dead → loads dark, which is safe). If a relay energises on its own at power-up, stop and check your wiring against ADR-0003/0009.

6Update wirelessly (OTA) — optional

Once the board is on WiFi and you’ve flashed it once over USB, you can update it without the cable:

pio run -e bench_ota -t upload

Set the device’s IP (from the boot log) as upload_port in the bench_ota environment if the .local mDNS name doesn’t resolve on your OS. OTA forces all relays off before it writes, so an interrupted flash can’t leave a coil energised. Windows OTA gotchas (extra network adapters, third-party AV firewalls) are covered in the README and ADR-0008.

7Troubleshooting

SymptomLikely cause / fix
No COM port appearsCP210x driver not installed, or a charge-only USB cable.
pio can’t find the projectYou opened the repo root — open firmware/ (where platformio.ini lives).
Console prints nothingWrong baud — set the monitor to 115200. Press Enter; type help.
WiFi line never appearsBad/blank credentials in secrets.h. The USB console still works — WiFi is optional on the bench.
LED heartbeats but no relay click(Full chain) check ULN COM→+12, the coil +12 rail (pin 86), and ULN GND; confirm inputs land on 13/25/26/27.
A relay is on at power-upWiring fault — relays must be fail-OFF. Recheck against ADR-0003 and the pull-downs on the ULN inputs.

8Where this fits

This is the manual, hands-on layer. On top of it runs the autonomous engine-run supervisor — SLEEP → ARMED → RUNNING ↔ POWERED (+ OFF_DELAY), the state machine in the electrical plan §13 / ADR-0012 — which drives this exact same RelayController. It's already implemented and bench-tested with simulated sensors (console: sim ign / sim vbus, state); the remaining milestone is the real INA3221 + PC817 sensing front-end (then sim real) and load-shedding. The bench console stays as a diagnostic tool underneath it.
